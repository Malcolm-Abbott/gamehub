## Repository Setup Guide

This guide explains how to build a repository **like this one** from scratch on a new machine. You can use it as a **linear walkthrough** any time you want to:

- Recreate this repo’s structure and tooling in a fresh folder.
- Start a brand‑new project that follows the same layout and conventions.
- Remind yourself of the “why” and “how” behind each step.

It is written assuming:

- You are on **Windows 10/11** (but notes for macOS/Linux are included where useful).
- You are comfortable typing commands into a terminal, but you do **not** need to be an expert.
- You want very explicit, step‑by‑step instructions and sanity checks.

> ---
>
> **Which terminal/shell can I use?**
>
> All the commands in this guide will work from common terminals:
>
> - On **Windows**: PowerShell, Command Prompt, or Git Bash
> - On **macOS**: Terminal (bash/zsh)
> - On **Linux**: your usual shell (bash, zsh, etc.)
>
> If a command fails in one terminal, try closing it and using another (for example, Git Bash on Windows) before troubleshooting further.
>
> ---
>
> **Important: don’t type the `<...>` placeholders**
>
> In this guide, any angle-bracket text like `<you>`, `<your-username>`, or `<project-name>` is a placeholder.
>
> Replace it with your own value, and do **not** include the `<` or `>` characters.
>
> Tip: `<you>` is your Windows username. For example, `C:/Users/<you>/Github` might look like `C:/Users/Sam/Github`.
>
> Example (placeholder format you must replace):
>
> ```bash
> cd C:/Users/<you>/Github
> git clone https://github.com/<your-username>/<project-name>.git
> cd <project-name>
> ```
>
> Example (this template’s real values):
>
> ```bash
> cd C:/Users/Sam/Github
> git clone https://github.com/Malcolm-Abbott/frontend-template.git
> cd frontend-template
> ```

## 1. What this repository template provides

- **Overall purpose**
  - A **front‑end project template** for React + TypeScript using **Vite**.
  - At the time this template was created, the **latest Vite versions** had some integration issues with **Tailwind CSS 4**, so this repo was built and tested specifically with **Vite 7**. Future versions of Vite may behave differently; if you upgrade, do it intentionally and test Tailwind carefully.
  - The example app is a **Todo List** with CRUD functionality, but you can replace it with any UI.

- **Repository layout (high level)**
  - Root folder (the Git repository itself).
  - `client/` – the actual React/Vite app.
  - `readme/` – documentation like deployment and setup guides.
  - Config files for linting, formatting, and deployment.

Below is an example of the important parts of the structure you’re aiming for (your actual file list may be a bit longer, but this is the essence):

```text
frontend-template/                # Repo root (Git repository)
  client/                         # Vite + React + TS app
    src/
      App.tsx
      main.tsx
      index.css
      App.css
      vite-env.d.ts
    index.html
    package.json
    package-lock.json
    tsconfig.json
    tsconfig.node.json
    vite.config.ts
    .eslintrc.cjs
    .gitignore

  readme/
    verceldeploy.md               # Vercel deployment instructions
    reposetup.md                  # (this file) repo setup instructions

  package.json                    # Root tooling: linting/formatting config packages
  package-lock.json
  .gitignore                      # Root Git ignore rules
  .eslintrc                       # Root ESLint config (if used)
  .prettierrc                     # Prettier formatting config
  .stylelintrc                    # CSS stylelint config
  .markuplintrc                   # HTML/markup linting config
  vercel.json                     # Vercel project configuration (optional)
  README.md                       # High‑level project README (optional)
```

When you “set up a repo like this”, the goal is to end up with a structure that broadly matches the above.

## 2. What you must have installed first (prerequisites)

You only need to do this once per machine. If these are already installed and working, you can skip to **section 3**.

- **Node.js and npm**
  - You need a recent **LTS** version of Node (for example Node 18+ or 20+/24+).
  - This gives you the `node` and `npm` commands.

- **Git**
  - Required so you can clone/push to GitHub and track your changes.

- **GitHub account**
  - Needed so you can host your repository remotely and later connect tools like Vercel.

If you are unsure whether you have these tools, follow the checks in **2.4**.

### 2.1. Install Node.js and npm (Windows)

1. Go to `https://nodejs.org/en`.
2. Download the **LTS (Long-Term Support)** installer for Windows (64‑bit).
3. Run the installer:
   - Accept the license.
   - Keep the default options.
   - Make sure **“Add to PATH”** (or similar wording) stays checked.
4. After installation, open a **new** terminal (PowerShell, Command Prompt, or Git Bash) and verify:

   ```bash
   node -v
   npm -v
   ```

   - Both commands should print version numbers (for example `v24.14.0`, `11.x.x`).
   - If you get “command not found” or a similar error, close the terminal, reopen it, and try again.

> **Note for Windows PowerShell users**
>
> If you see an error like:
>
> `npm : File 'C:\Program Files\nodejs\npm.ps1' cannot be loaded because running scripts is disabled on this system.`
>
> This is a **PowerShell execution policy** issue, not a Node problem.
>
> Easiest options:
> - Run `npm` from **Git Bash** or **Command Prompt** instead of PowerShell, or
> - (Advanced) In an elevated PowerShell (Run as Administrator) run:
>
>   ```powershell
>   Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
>   ```
>
> If you are not comfortable with this, simply use Git Bash or Command Prompt when working with Node/npm.

### 2.2. Install Git

1. Go to `https://git-scm.com/downloads`.
2. Download and run the installer for your OS.
3. Keep default options unless you know you need something else.
4. After installation, verify:

   ```bash
   git --version
   ```

   - You should see a version number.

### 2.3. Sign up for GitHub

1. Visit `https://github.com`.
2. Create an account or sign in.
3. (Recommended) Verify your email so you can create repositories and use all features.

### 2.4. Quick “do I have everything?” check

In a new terminal window, run:

```bash
node -v
npm -v
git --version
```

- If **all three** print version numbers, you are ready to continue.
- If any command fails, fix that tool first (sections **2.1** and **2.2**) before moving on.

## 3. Decide what you are building

There are two common scenarios where you might use this guide:

- **Scenario A – New repo from scratch**
  - You want to create a brand‑new repository that copies this template’s structure and behavior, but for a different app.

- **Scenario B – Recreating this exact repo elsewhere**
  - You want to rebuild this repository (same name/structure) on another machine or location, as a learning exercise.

The practical steps are almost the same. The only differences are:

- The **GitHub repo name** (e.g. `frontend-template`, `GameHub`, `TodoApp`, etc.).
- The **local folder name**.

In the rest of this guide, we will use a placeholder:

- `<project-name>` – the name of your new repository (for example `frontend-template`, `GameHub`, or `my-todo-app`).
- `<you>` – your Windows username (for example `C:/Users/<you>/Github`).

Whenever you see these placeholder values, substitute your own.

## 4. Create a new GitHub repository

These steps create the **empty GitHub repo** that will host your code.

1. Go to `https://github.com` and log in.
2. In the top‑left or top‑right, click **“+” → “New repository”** (or a **“New”** button).
3. Fill out the form:
   - **Repository name**: `<project-name>` (for example `GameHub`).
   - **Description**: optional, something like “React + Vite + TypeScript frontend”.
   - **Visibility**: choose **Public** or **Private** as you prefer.
   - **Initialize this repository with**:
     - You can leave these **unchecked** (no README, no .gitignore, no license) if you prefer to fully control everything locally.
     - Or you can let GitHub create a basic `README.md` and `.gitignore` – you can still adjust them later.
4. Click **“Create repository”**.

GitHub will now show you a page with instructions for pushing an existing repo or cloning. We will use **cloning** and build the structure locally.

## 5. Prepare your local project folder

You now need a folder on your machine where this repository will live.

### 5.1. Choose a base “projects” directory

You only need to decide this once. A common choice on Windows is:

- `C:/Users/<you>/Github`

If that folder doesn’t exist yet:

```bash
cd C:/Users/<you>
mkdir Github
cd Github
```

The idea is:

- `Github/` contains all your local GitHub repositories.
- Each project gets its own sub‑folder, named after the repo.

### 5.2. Clone the empty GitHub repository

From your chosen projects directory (for example `C:/Users/<you>/Github`), run:

```bash
cd C:/Users/<you>/Github
git clone https://github.com/<your-username>/<project-name>.git
cd <project-name>
```

You should now be inside your new repo folder. You can check:

```bash
pwd        # Git Bash / macOS / Linux
```

or simply look at your prompt in PowerShell/Command Prompt. It should end with `<project-name>`.

At this point your local folder is a **Git repository** linked to GitHub, but it is almost empty.

## 6. Create the project structure (like this template)

In this section you will:

- Create the `client/` folder using **Vite + React + TypeScript**.
- Add a `readme/` folder for documentation.
- Add root‑level configuration files that match this template’s behavior.

### 6.1. Create the `client/` app with Vite

From the **repo root** (`C:/Users/<you>/Github/<project-name>`):

1. Run the Vite project creation command, **pinning to Vite 7** and using the React + TypeScript template:

   ```bash
   npm create vite@7 client -- --template react-ts
   ```

   > **Why @7 and not @latest?**
   > At the time this repo was built, the newest Vite releases had some rough edges with **Tailwind CSS 4**. Using `@7` keeps you on the version that is known to work with the Tailwind setup in this template. You can experiment with newer versions later if you want, but start from a known‑good baseline.

   - This command:
     - Creates a `client` folder.
     - Uses the **React + TypeScript** starter template automatically (no prompts).

2. When the Vite tool completes, install dependencies:

   ```bash
   cd client
   npm install
   ```

3. Confirm the development server works:

   ```bash
   npm run dev
   ```

   - The terminal will show a local URL, usually `http://localhost:5173`.
   - Open it in your browser and make sure the starter Vite React page loads.

4. Stop the dev server:
   - Press `Ctrl + C` in the terminal window where it is running.

At this point, your `client/` folder should look similar to:

```text
client/
  index.html
  package.json
  package-lock.json
  tsconfig.json
  tsconfig.node.json
  vite.config.ts
  .eslintrc.cjs (or similar)
  .gitignore
  src/
    main.tsx
    App.tsx
    index.css
    App.css
    vite-env.d.ts
```

### 6.2. Align `client/package.json` scripts with this template

The Vite generator already provides the scripts you need, and they should closely match the ones used here:

- **`npm run dev`** – start local development server.
- **`npm run build`** – create production build (used by Vercel).
- **`npm run preview`** – preview the production build locally.
- Optionally **`npm run lint`** – run ESLint based on `client/.eslintrc.cjs`.

You can open `client/package.json` to verify the scripts exist. If you are using the same versions as this repo, you will see entries similar to:

- `"dev": "vite"`
- `"build": "tsc && vite build"`
- `"preview": "vite preview"`
- `"lint": "eslint . --ext ts,tsx --report-unused-disable-directives --max-warnings 0"`

You rarely need to edit these manually; they are just here so you know what they do.

### 6.3. Add Tailwind CSS (optional, if you want to match this repo)

This template uses **Tailwind CSS 4 with the Vite plugin**, as shown in `client/package.json`:

- Dependencies include `tailwindcss` and `@tailwindcss/vite`.
- The setup has been tested specifically with **Vite 7**; if you upgrade Vite beyond that, double‑check Tailwind’s docs and verify that `npm run dev` and your styles still behave as expected.

If you want your new repo to match that setup, follow the official Tailwind + Vite instructions (or mirror what is in this `client` folder). If you don’t need Tailwind, you can skip this and just use regular CSS.

> Since Tailwind setup can change over time, the most reliable source is the Tailwind docs for your version. The key idea is: if your `package.json`, `App.tsx`, and CSS files look like this repo’s `client/` directory, you’re aligned.

### 6.4. Create the `readme/` folder

Back in the **repo root** (not `client/`):

```bash
cd C:/Users/<you>/Github/<project-name>   # ensure you are at repo root
mkdir readme
```

Inside `readme/`, you can place:

- `reposetup.md` – this file, or your customized version of it.
- `verceldeploy.md` – deployment instructions like the ones in this repository.

You can copy the content of the existing `verceldeploy.md` (if you are using this repo as a reference) and adjust it for your new project.

### 6.5. Add root‑level tooling and config files

The root `package.json` in this template focuses on **tooling** (linting/formatting), not on running the app directly. The app runs from `client/`. A typical root `package.json` in this setup might contain:

- Dev dependencies such as:
  - `eslint`, `@typescript-eslint/*`, `prettier`
  - `stylelint`, `stylelint-config-*`
  - `markuplint`
  - `typescript`

To mirror this behavior:

1. From the repo root, ensure you have a `package.json`:

   ```bash
   cd C:/Users/<you>/Github/<project-name>
   npm init -y
   ```

2. Install the tooling packages you want (you can copy the list from this template’s root `package.json`):

   ```bash
   npm install --save-dev \
     @typescript-eslint/eslint-plugin @typescript-eslint/parser \
     eslint eslint-config-prettier eslint-config-standard \
     eslint-plugin-html eslint-plugin-import eslint-plugin-node eslint-plugin-promise \
     markuplint npm-run-all prettier \
     stylelint stylelint-config-recommended stylelint-config-standard \
     typescript
   ```

3. Add configuration files in the root:
   - `.eslintrc`
   - `.prettierrc`
   - `.stylelintrc`
   - `.markuplintrc`
   - `.gitignore`

You can use the versions in this repository as a starting point and tweak them later. The exact rules are flexible; the important part is that:

- You have a **consistent linting/formatting setup**.
- Your tooling is installed in the root so your editor can pick it up.

## 7. Run, commit, and push

By this point you should have:

- A Git repository at `C:/Users/<you>/Github/<project-name>`.
- A `client/` app created by Vite (React + TypeScript).
- A `readme/` folder with at least `reposetup.md` (and optionally `verceldeploy.md`).
- Root tooling set up via `package.json` and config files.

Now you will:

1. **Test the app locally.**
2. **Commit the initial version.**
3. **Push to GitHub.**

### 7.1. Test the app locally from `client/`

From the repo root:

```bash
cd C:/Users/<you>/Github/<project-name>
cd client
npm install          # if not already done
npm run dev
```

- Open the URL shown in the terminal (usually `http://localhost:5173`).
- Confirm the app loads and behaves as expected (at least the starter Vite page).

When you’re done, stop the dev server with `Ctrl + C`.

### 7.2. Check Git status

From the **repo root**:

```bash
cd C:/Users/<you>/Github/<project-name>
git status
```

You should see a list of untracked files (for example `client/`, `readme/`, and your config files).

### 7.3. Make the initial commit

Add everything and commit:

```bash
git add .
git commit -m "Initial project setup (Vite + React + TS)"
```

If Git complains about your name/email not being set, it will show you the commands to configure them, for example:

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```

Run those once, then repeat the commit.

### 7.4. Push to GitHub

Push your local changes to the remote repository:

```bash
git push origin main
```

or, if your default branch is `master`:

```bash
git push origin master
```

After this completes without errors:

- Go to `https://github.com/<your-username>/<project-name>`.
- You should see all your files, including `client/` and `readme/`.

Your repository is now:

- **Fully set up** with React + TypeScript + Vite in `client/`.
- **Configured** with root‑level tooling (linting/formatting).
- **Ready** for deployment (see `readme/verceldeploy.md` for Vercel instructions).

## 8. Quick checklist for recreating this repo

Use this whenever you want to build a similar repo from scratch on a new machine. You can simply tick each line off as you go.

1. **Prerequisites**
   - Node.js + npm installed (`node -v`, `npm -v` work).
   - Git installed (`git --version` works).
   - GitHub account ready.

2. **Create GitHub repository**
   - New repo at `https://github.com/<your-username>/<project-name>`.
   - Clone it into `C:/Users/<you>/Github/<project-name>`.

3. **Create project structure**
   - From repo root, run `npm create vite@7 client -- --template react-ts` (to match this template’s tested Vite + React + TS setup).
   - `cd client && npm install`.

4. **Verify app runs**
   - `npm run dev` inside `client/`.
   - Visit `http://localhost:5173` and confirm it loads.

5. **Add docs and tooling**
   - Create `readme/` folder.
   - Add `reposetup.md` and (optionally) `verceldeploy.md`.
   - In the repo root, `npm init -y`.
   - Install dev dependencies for linting/formatting (mirror this template’s `package.json` if desired).
   - Add root configs: `.eslintrc`, `.prettierrc`, `.stylelintrc`, `.markuplintrc`, `.gitignore`.

6. **Commit and push**
   - From repo root: `git add .`
   - `git commit -m "Initial project setup (Vite + React + TS)"`
   - `git push origin main`

7. **(Optional) Deploy**
   - Follow `readme/verceldeploy.md` to connect your repo to Vercel and deploy.

With these steps, you should be able to:

- Take a completely blank machine.
- Install required tools once.
- Build a repo that looks and behaves like this template.
- Run it locally and, if you choose, deploy it using the existing Vercel guide.
